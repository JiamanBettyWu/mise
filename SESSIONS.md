## 2026-07-04 (structured outputs on trip_plan)

Shipped **[#123](https://github.com/JiamanBettyWu/wardrobe-ai/issues/123) → [PR #130](https://github.com/JiamanBettyWu/wardrobe-ai/pull/130)**: the structured-outputs pilot on the `trip_plan` call. `recommend_packing_plan` now goes through `create_tracked_parsed` in `services/claude.py`, which enforces `schemas.PackingPlanOutput` via the API's `output_config.format` — the shape is guaranteed by construction, so the #120 gap-dropping loop, the #122 missing-key defaults, and the "Return ONLY a JSON object…" block in `PACKING_SYSTEM_PROMPT` are all deleted. **Pre-merge review surfaced two findings, both fixed before merge:** (1) the first cut called `messages.parse()`, which validates *inside* the SDK call — a paid request whose output failed validation (e.g. truncation at `max_tokens`) would raise before `_record_usage`, silently skipping the `llm_usage` row that #114 metering depends on; (2) the caller indexed `response.content[0].parsed_output`, which breaks on an empty or non-text first content block. The fix for both: `create_tracked_parsed` decomposes what `parse()` does into its three steps — `transform_schema` → `messages.create(output_config.format=…)` → validate — with usage recording *between* the call and the validation, and returns the validated `PackingPlanOutput` directly, so callers never touch response internals; a validation failure raises a `ValueError` carrying `stop_reason`, restoring the truncation diagnostics `parse_json` used to give. One accepted trade-off: `transform_schema` is imported from a private SDK path (`anthropic.lib._parse._transform`) — it's the exact helper `messages.parse()` uses, and the offline suite imports the module, so an SDK relocation fails loudly in CI rather than silently. Pre-flight from the issue all checked out (model support, schema restrictions, live end-to-end call). Suite 221 → 222. The paid `--trials 3` before/after eval was deliberately skipped — run it later if before/after variance numbers are wanted.

## 2026-07-02 (bbox feasibility experiment)

Ran the **Claude bbox feasibility experiment** ([#96](https://github.com/JiamanBettyWu/wardrobe-ai/issues/96) → [PR #97](https://github.com/JiamanBettyWu/wardrobe-ai/pull/97), merged; issue closed with the verdict) — the gate for B-full cropped per-item thumbnails, deferred out of #24. New checked-in harness `backend/scripts/bbox_experiment.py`: asks the production tagging model for per-item bounding boxes, draws them with Pillow, saves annotated copies to gitignored `bbox_annotated/`. **Verdict: feasible** — across 5 real photos / 16 items (jewelry spread, two-garment flat-lay, light-touch + sunglasses, real occlusion with a vest covering ⅓ of a sweater, and a 5-item spread): 16/16 identified and correctly typed, zero merged boxes, zero missed items. Two findings mid-run, both now in the script: (1) the first run looked like a *negative* result — every box shifted up-left and ~25% small — but the errors were systematic, not random, which was the tell: the **vision API silently downscales images over ~1.15 MP / 1568px long edge before the model sees them**, so the model reported coordinates in its resized frame; pre-resizing below those limits ourselves makes model space pixel-identical to our bytes (→ LEARNINGS). Extended the **Weave tracing track** ([#85](https://github.com/JiamanBettyWu/wardrobe-ai/issues/85) — still Phase 1, tracing only) from the daily recommender to the **trip-planner LangGraph** ([PR #93](https://github.com/JiamanBettyWu/wardrobe-ai/pull/93)). Key finding, confirmed against the *current* Weave docs (not memory — #85 guardrail): unlike the recommender (plain straight-line Python, which needed four hand-placed `@weave.op`), the trip planner is a LangGraph — a LangChain `Runnable` under the hood — and **Weave's LangChain integration auto-traces the whole graph the moment `weave.init()` runs** (init sets `WEAVE_TRACE_LANGCHAIN=true` and installs a callback handler that fires on every runnable step). So the entire node tree (`get_weather`, `infer_weather_if_needed`, `reason_and_select`, `plan_purchase_queries`, `search_purchases`, `generate_output`) plus the auto-patched Anthropic calls render **with zero decorators and zero changes to `trip_planner.py`** — verified live in the Weave UI (node names show up cleanly; the `has_gaps` branch is visible). Tracing lives entirely in a new manual launcher **`jobs/trace_trip.py`** (sibling of `trace_daily.py`): it `init_weave()`s before importing the graph — critical, since `trip_planner` compiles the graph and builds the Claude client at module load, so both the Anthropic autopatch and the LangChain callback must already be installed — then invokes `run()` on a sample trip. The fail-closed guarantee is *stronger* than the recommender's: `trip_planner.py` doesn't import weave at all (not even the shim), so the graph module **structurally cannot** trace on Render or the `/trip` router; running the launcher is the sole opt-in. One polish: the auto-trace root defaulted to the integration's generic `langchain.Chain.LangGraph`, so the launcher wraps the invoke in a named **`@op(name="plan_trip")`** (via the same `observability` shim, real `weave.op` after init) — the top-level call now reads `plan_trip` and the LangGraph subtree nests underneath it. ⚠️ Unlike `trace_daily.py` there's **no `persist=False` lever** — the planner writes nothing to the DB, but every run calls Claude and `search_purchases_node` hits **SerpAPI live** (real quota), so a trace run is a real paid run. **Manual reproduce:** `WANDB_API_KEY` in repo-root `.env`, then `uv --project backend run python jobs/trace_trip.py`. **Next:** still Phase 2 — `weave.Evaluation` over 👍/👎 history (code-based scorers first, LLM-as-judge on a different model family for the fuzzy axis).
